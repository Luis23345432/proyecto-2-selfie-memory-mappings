2. Arquitectura general

La implementación agrega una ruta nueva: desde un proceso se llega a una página de archivo mediante metadata de mappings, page table y page cache global.

5. mapping_entry

Cada mapping activo tiene una entrada de 8 palabras. Es metadata, no contiene los bytes del archivo; los bytes viven en cache frames.

OffsetCampoValor ejemploExplicación
0nextNULLSiguiente mapping de la lista del proceso.
1addr0x10000000Inicio virtual del mapping.
2original_length3000Longitud exacta solicitada por el usuario.
3rounded_length4096Longitud redondeada a páginas completas.
4prot0 READPermisos: 0 read, 1 write, 2 read/write.
5fd3Descriptor de archivo usado como file_id simplificado.
6file_offset0Offset inicial dentro del archivo.
7flagsACTIVEMarca si la entrada sigue activa.
Por qué existe original_length: si el usuario pidió 3000 bytes, msync debe saber que la última página no debe escribirse completa si no corresponde.
Por qué existe rounded_length: la MMU trabaja con páginas; 3000 bytes ocupan una página completa de 4096 bytes en la page table.

6. Page Cache

El page cache es global. Está implementado como una lista enlazada de page_cache_entry. Cada entrada identifica una página de archivo mediante file_id y file_page, apunta a un cache frame y guarda estado de dirty, refcount y LRU.

CampoQué guardaPara qué sirve
nextSiguiente entrada de la lista.Recorrer el page cache global.
file_idID estable asignado al path abierto.Compartir cache entre dos fd del mismo archivo.
file_pageNúmero de página dentro del archivo.Ubicar qué bloque de archivo está cacheado.
cache_frameDirección del frame global.Contenido real que usa la page table.
dirty0 limpio, 1 modificado.Decidir si msync debe hacer pwrite.
refcountCantidad de mappings usando la entrada.No reemplazar páginas todavía usadas.
fdDescriptor abierto más reciente.Hacer pread/pwrite.
last_usedMarca de uso reciente.Elegir víctima LRU cuando se llenan los frames.
flagsReservado.Extensión futura.
PreguntaRespuesta según la implementación
Cómo se busca una página?El fd se traduce a file_id y se busca (file_id, file_page).
Qué significa dirty?El frame cambió en memoria y todavía no fue escrito con msync.
Qué significa refcount?Cuántos mappings/procesos apuntan a ese frame.
Cuándo se reutiliza un frame?Si ya existe la entrada, se comparte; si el cache está lleno, LRU elige una entrada con refcount=0.
 
flowchart LR
  A[mmap necesita fd=3 page=0] --> X[fd -> file_id por path]

  B -->|Si| C[Reusar cache_frame]
  B -->|No| D{Hay frame libre?}
  D -->|Si| E[allocate_cache_frame]
  D -->|No| L[evict LRU con refcount=0]
  E --> F[pread archivo -> frame]
  L --> F
  F --> G[crear page_cache_entry]
  C --> H[touch last_used + refcount]
  G --> H

17. Mejoras recientes del page cache

Después de la primera versión funcional se agregaron dos mejoras puntuales: identidad estable por path para compartir cache entre fd distintos, y reemplazo LRU simple para reutilizar cache frames.

Identidad fd -> file_id

Cuando openat abre un archivo, se registra el path en una tabla interna. Si el mismo path se abre otra vez, el nuevo fd reutiliza el mismo file_id.

fd1 = open("archivo.txt", ...)
fd2 = open("archivo.txt", ...)
fd1 != fd2
file_id(fd1) == file_id(fd2)
Esto permite que dos mappings del mismo archivo compartan la misma entrada del page cache.

LRU simple

Cada entrada guarda last_used. Si no quedan cache frames libres, se busca una entrada con refcount=0 y se reemplaza la menos usada recientemente.

si no hay frame libre:
  victima = menor last_used con refcount == 0
  si dirty: pwrite(...)
  reutilizar frame
No se reemplaza una página que todavía está mapeada por algún proceso.
CambioArchivo / funciónImpacto
Page cache entry de 9 palabrasPAGE_CACHE_ENTRY_SIZEAgrega fd y last_used.
Registro de path abiertoregister_file_identity en openatDos fd del mismo path comparten file_id.
Busqueda por file_idfind_page_cache_entryEl page cache ya no depende directamente del número de fd.
Reemplazo LRUevict_page_cache_entryPermite superar los 256 frames reutilizando entradas no referenciadas.

19. Comparación con Linux real

AspectoSelfie simplificadoLinux real
mmapCrea mapping inmediato y PTEs hacia cache frames.Puede usar carga perezosa por page faults.
munmapQuita mapping y no persiste.También desmapea; la persistencia depende de flags y writeback.
msyncÚnico mecanismo de persistencia del proyecto.Fuerza sincronización según flags y políticas del kernel.
Page cacheLista global con identidad por path, refcount, dirty y LRU simple.Subsistema complejo con radix/xarray, writeback, LRU, locks e identidad por inode/archivo.
Permisos0=READ, 1=WRITE, 2=READ_WRITE.Flags bitmask: PROT_READ, PROT_WRITE, etc.
File descriptorsSe traduce fd -> file_id usando el path registrado en openat.Usa estructuras de archivo/inodo y objetos globales.
Lazy loadingNo es el foco; se carga al mapear.Normalmente se carga bajo demanda.
Dirty pagesDirty bit explícito por page cache entry.Dirty tracking integrado con VM y page cache.
ForkMappings comparten cache frames; memoria normal se copia.Usa copy-on-write y VMAs compartidas/privadas.

20. Errores comunes

ErrorPor qué pasaCómo evitarlo
Pensar que munmap guarda cambios.Se asocia "cerrar" con "guardar".Recordar: solo msync hace pwrite.
Confundir page cache con page table.Ambas hablan de páginas.Page table traduce; page cache guarda contenido de archivo.
Creer que fork duplica cache frames.La memoria normal sí se copia.En mmap, hijo y padre apuntan al mismo frame.
Olvidar refcount.El frame parece pertenecer a un solo proceso.Mirar cuántos mappings apuntan al mismo entry.
Creer que dirty significa archivo actualizado.Dirty suena a "ya cambió".Dirty significa "cambió en memoria, falta msync".
Creer que dos fd siempre significan dos archivos distintos.Un mismo path puede abrirse varias veces.En este proyecto se registra fd -> file_id por path; Linux real usaría inode/archivo.
Reemplazar una página todavía usada.LRU sin refcount sería inseguro.Solo elegir víctimas con refcount=0.

mmapCrea la relación entre memoria virtual y archivo.mmap(0,4096,0,fd,0)
mapping_entryDescribe el rango virtual mapeado.addr=0x10000000,length=4096
Page TablePermite acceder al frame desde una dirección virtual.VP -> F0
Page CacheGuarda entradas globales por archivo y página.file_id=1,page=0
Cache FrameContiene una página del archivo.F0 = bytes 0..4095
DirtyMarca cambios pendientes de guardar.dirty=1 después de write.
RefcountCuenta cuántos mappings usan un frame.2 después de fork.
forkComparte cache frames entre padre e hijo.Padre e hijo apuntan a F0.
LRUReutiliza frames cuando el cache se llena.Víctima con refcount=0.
msyncPersiste cambios dirty al archivo.pwrite(F0).
mmap crea la relación, page cache guarda las páginas, la page table permite accederlas, fork las comparte, LRU reutiliza frames y msync persiste los cambios.
